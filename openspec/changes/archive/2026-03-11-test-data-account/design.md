## Context

E.A.S.Y 育儿助手应用需要一套测试数据注入机制，用于开发测试阶段验证各功能模块。当前每次测试都需要手动创建数据，效率低下且测试覆盖不全。

**约束条件**：
- 测试数据账号仅在开发/测试环境使用，不影响生产用户
- 数据注入后应可清除，支持重复测试
- 复用现有数据库表结构，不新增表

## Goals / Non-Goals

**Goals:**
- 提供一键登录测试账号的能力
- 自动注入完整、真实的测试数据
- 数据覆盖核心测试场景（多宝宝、长时间跨度、多种活动类型）
- 清晰标识测试数据账号状态

**Non-Goals:**
- 生产环境的测试功能
- 远程测试数据服务器同步
- 自定义测试数据配置界面

## Decisions

### 1. 测试数据账号实现方式

**决定**：使用本地标识 + 预设数据注入

**原因**：
- 简单直接，无需后端服务
- 测试数据固定，便于验证功能
- 快速开发，立即可用

**替代方案**：
- 真实的测试账号服务端登录 → 需要后端支持，当前阶段不可行
- 从 JSON 文件导入 → 需要额外文件管理，增加复杂度

### 2. 测试数据生成策略

**决定**：硬编码预设数据 + 动态日期偏移

**原因**：
- 测试数据内容固定，便于断言验证
- 日期使用相对偏移（如"今天-1天"），保证数据时效性
- 不依赖外部文件，代码即文档

**数据设计**：
- 宝宝1：3个月大，有完整90天数据
- 宝宝2：8个月大，有混合喂养数据
- 时间线数据：覆盖 Eat/Activity/Sleep/Poop 全类型
- 疫苗记录：按月龄匹配疫苗库

### 3. 测试数据入口位置

**决定**：在"我的"页面底部添加测试数据登录按钮

**原因**：
- "我的"页面是设置和账户相关功能的聚集地
- 不影响主要功能页面的用户体验
- 便于后续扩展（如清除测试数据）

### 4. 状态管理

**决定**：使用 Riverpod StateNotifier 管理测试数据状态

**原因**：
- 与现有架构一致
- 状态变化可监听，UI 可响应式更新
- 便于扩展（如显示注入进度）

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 测试数据账号误用于生产 | 明确标识，登录时显示警告提示 |
| 测试数据影响真实用户数据 | 使用独立的测试用户 ID，便于筛选清除 |
| 数据量过大影响性能 | 控制数据量，90天数据约 500-800 条记录 |

## Migration Plan

无需迁移，纯新增功能。

**部署步骤**：
1. 新增 `TestDataService` 和 `TestDataProvider`
2. 修改 `ProfilePage` 添加入口
3. 测试验证

**回滚**：移除相关代码即可

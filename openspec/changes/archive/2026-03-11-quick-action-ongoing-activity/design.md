## Context

当前首页快捷操作栏的按钮长按功能是弹出"快捷记录弹窗"，用于新增一条已完成的历史记录。这与实际育儿场景不符——父母通常是在活动**开始**时长按按钮（如"现在开始喂奶"），而不是在活动结束后去补录。

本项目使用 Flutter + Riverpod + Drift 技术栈，采用 Provider 直接访问数据库的架构（无 Repository 层）。

## Goals / Non-Goals

**Goals:**
- 长按快捷按钮创建"正在进行中"的活动记录（无结束时间）
- 在时间线上清晰展示"进行中"与"已完成"的活动状态
- 支持点击"结束"按钮完成正在进行的活动
- 正在进行的活动卡片需要有明显的视觉区分

**Non-Goals:**
- 不支持多个活动同时进行（同一时间只能有一个进行中活动）
- 不支持暂停/恢复功能
- 本次不涉及首页计时器的改动

## Decisions

### Decision: 数据模型设计

**选择**: 在 `ActivityRecords` 表新增 `status` 字段（0=进行中, 1=已完成），`endTime` 改为可空。

**理由**:
- 使用状态字段比仅依靠 `endTime` 是否为空更清晰，便于扩展（未来可能支持暂停状态）
- 保持向后兼容，现有数据默认为已完成状态

**替代方案**: 仅用 `endTime` 可空来表示进行中
- 拒绝原因：语义不够明确，查询时需要理解业务逻辑

### Decision: 快捷按钮长按行为

**选择**: 长按创建进行中活动，短按保持计时器功能不变。

**理由**:
- 符合用户直觉：长按 = "我要开始做某事"
- 短按计时器功能已在其他模块实现，本次不改动

**替代方案**: 长按弹出选择菜单（开始计时 / 新增记录）
- 拒绝原因：增加操作步骤，违背快捷操作的设计初衷

### Decision: 进行中活动状态管理

**选择**: 使用独立的 `ongoingActivityProvider` 来管理当前进行中的活动。

**理由**:
- 与 timelineProvider 分离，避免频繁刷新整个时间线列表
- 支持跨页面监听正在进行的活动（如在其他页面也能看到结束按钮）

### Decision: 同一时间单活动限制

**选择**: 同一宝宝同一时间只能有一个进行中活动。

**理由**:
- 简化状态管理
- 符合 E.A.S.Y. 育儿法的实际场景（吃、玩、睡是顺序关系，不会同时进行）
- 防止用户误操作导致数据混乱

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 用户可能误触长按，意外创建进行中活动 | 添加长按动画反馈（缩放效果），让用户明确知道触发了长按 |
| 进行中活动忘记结束，导致统计数据异常 | 在次日时间线顶部显示"昨日有未结束的活动"提醒 |
| 与现有计时器功能概念冲突 | 在首页添加明确的视觉区分：计时器 vs 进行中活动卡片 |

## Migration Plan

1. 数据库迁移：新增 `status` 字段，默认值为 1（已完成），现有 `endTime` 不为空的记录保持现状
2. 代码适配：timelineProvider 查询时需要过滤或特殊处理进行中活动
3. 功能渐进：先实现长按创建进行中活动，再实现结束功能

## Open Questions

- 进行中活动的专属字段（如吃奶的奶量）是否允许在创建时填写，还是仅在结束时填写？
- 是否需要在创建进行中活动时选择具体子类型（如母乳/配方奶）？

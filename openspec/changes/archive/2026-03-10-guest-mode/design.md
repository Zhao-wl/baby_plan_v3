## Context

当前应用需要用户注册才能使用，这增加了使用门槛。根据探索讨论，我们确定了以下设计方向：

- 采用方案 B：创建游客用户记录（而非跳过 Users 表）
- 游客用户可随时升级为正式账号
- 游客数据仅限本设备
- 升级逻辑作为占位，后续迭代实现

## Goals / Non-Goals

**Goals:**
- 新用户首次启动自动创建游客身份，直接进入首页
- 游客可使用完整离线功能（宝宝、活动记录、计时器、疫苗等）
- 游客状态在 UI 中清晰展示
- 提供登录/升级入口（占位实现）
- 为后续升级逻辑预留扩展点

**Non-Goals:**
- 升级为正式账号的完整实现（后续迭代）
- 多用户切换功能
- 游客数据跨设备同步

## Decisions

### D1: 游客用户数据模型

**决策**：在 Users 表新增 `isGuest` 字段标识游客身份。

**理由**：
- 数据模型统一，所有宝宝/活动记录都有 userId 归属
- 升级逻辑简单：更新现有用户记录而非迁移数据
- 用户可自定义游客昵称，体验更连贯

**结构**：
```dart
class Users extends Table {
  // ... 现有字段 ...
  BoolColumn get isGuest => boolean().withDefault(const Constant(false))();
}
```

### D2: AuthProvider 架构

**决策**：新增 `AuthProvider` 管理用户认证状态。

**职责**：
- 启动时检查/创建当前用户
- 提供当前用户状态流
- 提供登录/登出/升级方法（占位）

**与其他 Provider 关系**：
```
databaseProvider
      │
      ▼
authProvider (新增)
      │
      ├─▶ babiesProvider (按用户过滤)
      ├─▶ familyProvider (游客时返回 null)
      └─▶ syncProvider (游客时禁用)
```

### D3: 首次启动流程

**决策**：App 启动时自动创建游客用户（如果不存在）。

**流程**：
```
main.dart 启动
    │
    ▼
AuthProvider 初始化
    │
    ├─▶ 检查本地用户
    │       │
    │       ├─▶ 有用户 → 使用现有用户
    │       │
    │       └─▶ 无用户 → 创建游客用户
    │                   nickname = "游客"
    │                   isGuest = true
    │
    ▼
进入 MainPage
```

### D4: 游客限制实现

**决策**：通过 Provider 层条件逻辑限制云端功能。

**限制项**：
| 功能 | 游客可用 | 实现方式 |
|------|---------|---------|
| 宝宝管理 | ✓ | - |
| 活动记录 | ✓ | - |
| 计时器 | ✓ | - |
| 疫苗记录 | ✓ | - |
| 云同步 | ✗ | syncProvider 返回禁用状态 |
| 家庭组 | ✗ | familyProvider 返回 null，UI 显示提示 |

## Risks / Trade-offs

### [风险] 用户误以为游客数据会自动同步
**缓解**：首页明确显示"游客模式"状态，点击可查看说明

### [风险] 游客数据丢失（设备损坏/卸载）
**缓解**：属于游客模式的预期行为，升级提示中说明

### [风险] 数据库迁移兼容性
**缓解**：`isGuest` 字段默认值为 `false`，现有数据不受影响

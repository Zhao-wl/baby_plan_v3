## Context

当前预测算法将全天所有时段的活动数据平均处理，忽略了时间段差异。这导致：
- 夜间预测不准确（夜间间隔更长）
- 无法识别"睡前喂奶"、"长觉"等规律
- 用户无法利用预测培养规律作息

现有架构：
```
PredictionService
    │
    ├── 查询最近7天活动记录（全局）
    ├── 计算历史模式（简单平均）
    ├── 查询月龄基准数据（无时段维度）
    ├── 加权融合（70% + 30%）
    └── 合并15分钟内预测
```

**约束**：
- 本地计算，无服务端支持
- 简单统计算法，不使用机器学习
- 需要兼容现有数据结构

## Goals / Non-Goals

**Goals:**
- 引入时间段划分（5个时段），历史数据按时段分组分析
- 引入分时段月龄基准数据
- 增加"距离上次睡眠时间"维度
- 夜间提供预测，支持作息培养指导
- 保持算法简单可解释

**Non-Goals:**
- 不引入机器学习模型
- 不实现云端同龄宝宝数据对比
- 不实现预测历史记录
- 不修改现有数据库表结构（仅扩展 JSON 数据格式）

## Decisions

### 1. 时间段划分：5 个固定时段

**决策**：采用 5 个固定时段，符合宝宝作息规律。

```
时段          时间范围         特点
──────────────────────────────────────────
早晨 morning   06:00-09:00    刚醒来，吃奶需求高
上午 forenoon  09:00-12:00    活动期
下午 afternoon 12:00-18:00    最长时段，包含午睡
傍晚 evening   18:00-22:00    睡前准备期
夜间 night     22:00-06:00    长觉期，间隔拉长
```

**理由**：
- 符合宝宝自然作息规律
- 时段数量适中，每个时段有足够样本
- 夜间单独划分，便于识别长觉模式

**备选方案**：按小时划分（24 个时段）
- 缺点：数据过于分散，每个时段样本不足

### 2. 睡眠维度：三阶段划分

**决策**：基于"距离上次睡眠结束的时间"划分三个阶段。

```
阶段              时间范围         行为特点
──────────────────────────────────────────────
刚醒期 awake-early   0-2小时       吃奶需求高
活动期 awake-mid     2-4小时       活动时间，排泄增多
疲劳期 awake-late    4+小时        即将入睡，睡眠预测置信度高
```

**实现**：
- 查询最近一次已完成的睡眠活动
- 计算睡眠结束时间到当前的时间差
- 根据时间差确定阶段，调整预测权重

**备选方案**：基于时钟时间
- 缺点：无法反映宝宝真实生理状态

### 3. 权重融合：三元加权

**决策**：采用三元加权算法（时段历史 + 全天历史 + 月龄基准）。

```
if (时段样本 >= 3):
    预测 = 时段历史 × 0.4 + 全天历史 × 0.3 + 月龄基准 × 0.3
elif (时段样本 >= 1):
    预测 = 时段历史 × 0.2 + 全天历史 × 0.5 + 月龄基准 × 0.3
else:
    预测 = 全天历史 × 0.7 + 月龄基准 × 0.3  // 回退到当前算法
```

**理由**：
- 时段样本充足时，时段数据权重更高
- 时段样本不足时，平滑回退到全天数据
- 月龄基准作为稳定底座

### 4. 数据存储：扩展现有 JSON 格式

**决策**：扩展现有 `age_activity_patterns.json`，增加时段维度。

```json
{
  "version": 2,
  "patterns": [
    {
      "week": 0,
      "activityType": 0,
      "global": {
        "intervalMinutes": 150,
        "durationMinutes": 25,
        "countPerDay": 10
      },
      "timeSlots": {
        "morning": { "intervalMinutes": 120, "durationMinutes": 30 },
        "forenoon": { "intervalMinutes": 150, "durationMinutes": 25 },
        "afternoon": { "intervalMinutes": 150, "durationMinutes": 25 },
        "evening": { "intervalMinutes": 180, "durationMinutes": 20 },
        "night": { "intervalMinutes": 240, "durationMinutes": 25 }
      }
    }
  ]
}
```

**理由**：
- 向后兼容：保留 global 字段作为默认值
- 无需修改数据库表结构
- 历史窗口扩展到 14 天以获取更多时段样本

### 5. 夜间预测：启用并增强提示

**决策**：夜间（22:00-06:00）启用预测，但增加特殊提示。

**提示策略**：
```
夜间预测提示：
- "夜间长觉，预计间隔 X 小时"
- "如需夜奶，建议在 X:XX 前完成"
- "宝宝可能睡到 X:XX，帮助培养整夜睡眠"
```

**理由**：
- 用户需要预测来培养规律作息
- 夜间不规律宝宝的父母更需要指导
- 提示帮助用户理解夜间模式

## Risks / Trade-offs

### Risk 1: 时段数据不足导致预测不稳定
**缓解**：
- 实现平滑回退机制（时段 → 全天 → 月龄基准）
- 历史窗口从 7 天扩展到 14 天
- 显示置信度，管理用户期望

### Risk 2: 时段边界活动归属模糊
**缓解**：
- 边界附近（±30分钟）的活动贡献给相邻两个时段
- 使用渐变权重，避免硬切换

### Risk 3: 数据格式变更导致旧版本不兼容
**缓解**：
- 保留 `global` 字段作为默认值
- 版本号升级，加载时检测版本
- 旧版本数据可正常使用（仅缺少时段精度）

### Risk 4: 睡眠数据缺失导致睡眠维度失效
**缓解**：
- 睡眠维度为可选增强，无睡眠数据时降级到时段模式
- 置信度标记数据来源

## Migration Plan

### 部署步骤

1. **数据迁移**：
   - 更新 `age_activity_patterns.json` 到 v2 格式
   - 应用启动时检测版本，自动加载新数据

2. **代码更新**：
   - 新增 `TimeSlot` 枚举和 `AwakeStage` 枚举
   - 重构 `PredictionService`，增加时段和睡眠维度计算
   - 修改 `prediction_provider.dart`，移除夜间禁用逻辑

3. **测试验证**：
   - 验证旧数据兼容性
   - 验证时段预测准确性

### 回滚策略

- JSON 文件回退到 v1 格式
- 代码保留全局预测逻辑作为回退路径
- 无数据库结构变更，无需数据迁移回滚

## Open Questions

1. **时段基准数据来源**：是否需要专业育儿顾问审核？当前数据基于通用育儿书籍整理。
2. **睡眠阶段阈值**：0-2小时 / 2-4小时 / 4+小时 的划分是否适合所有月龄？可能需要按月龄调整。
3. **提示文案**：夜间预测的具体提示文案需要用户测试验证。

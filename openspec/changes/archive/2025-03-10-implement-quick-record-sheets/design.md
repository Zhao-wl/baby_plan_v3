## Context

当前项目已有成熟的`HalfScreenSheet`通用半屏弹窗组件，支持滑入动画、下拉关闭、未保存数据二次确认等基础功能。快捷操作栏已实现4种活动类型（吃奶/玩耍/睡眠/排泄）的计时功能，数据库层`ActivityRecords`表已包含所有需要的字段，可直接支持详细记录的存储。本次设计将基于现有能力扩展，无需从零构建基础组件。

## Goals / Non-Goals

**Goals:**
1. 实现可复用的快捷记录弹窗组件，支持4种活动类型的专属字段动态渲染
2. 保持与现有系统UI风格一致，遵循已有的设计规范
3. 兼顾两种使用场景：快速记录（仅时间）和详细记录（完整字段）
4. 表单数据直接映射到数据库表结构，无需额外转换层
5. 支持两种触发方式：长按快捷按钮触发、计时结束自动触发

**Non-Goals:**
1. 不实现复杂的表单校验逻辑，仅做必填项基础校验
2. 不修改现有数据库表结构，完全复用现有字段设计
3. 不实现历史记录编辑功能，仅处理新增记录场景
4. 不支持多语言，保持简体中文界面

## Decisions

1. **弹窗容器选择**：直接复用现有`HalfScreenSheet`通用组件
   - 理由：已有成熟实现，包含下拉关闭、未保存确认、响应式适配等功能，减少重复开发
   - 备选方案：自定义Modal组件 → 放弃，重复造轮子且无法保证体验一致性

2. **表单渲染方案**：采用动态字段渲染架构
   - 理由：4种活动类型的字段差异较大，动态渲染可减少重复代码，便于后续扩展新活动类型
   - 实现：根据传入的`ActivityType`参数，动态生成对应字段的表单控件

3. **保存模式设计**：提供双模式保存
   - 快速保存：仅记录开始/结束时间，其余字段留空，1秒完成录入
   - 详细保存：填写所有字段后保存，保证数据完整性
   - 理由：兼顾不同用户的使用习惯，满足快速操作和详细记录的双重需求

4. **触发方式设计**：支持两种触发入口
   - 长按快捷操作栏按钮：直接弹出对应类型的空白表单
   - 计时结束后自动弹出：携带已记录的时间信息，引导用户补充详情
   - 理由：覆盖不同使用场景，提升操作流畅度

## Risks / Trade-offs

| 风险 | 缓解方案 |
|------|---------|
| 表单字段过多导致移动端录入效率低 | 默认仅展开必填字段，非必填字段折叠显示，优先保证快速操作 |
| 不同活动类型字段差异大导致组件复杂度高 | 按活动类型拆分子组件，模块化实现，降低耦合度 |
| 用户可能误触弹窗关闭丢失数据 | 启用未保存数据检测，关闭时自动弹出二次确认对话框 |
| 表单样式与现有系统不一致 | 复用现有表单样式组件（输入框、选择器、按钮等），保持视觉统一 |

### 权衡决策
- 为保证移动端操作流畅度，适当牺牲桌面端的复杂排版灵活性
- 优先保证新增记录的性能，暂不考虑编辑场景的复杂度
